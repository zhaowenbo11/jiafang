# 门店实时选布成品预览系统 开发任务拆解

## 1. 目标

本任务拆解面向一期 MVP，目标是在最短周期内完成一个可到店演示、可真实试运行的版本。

MVP 必须交付：

- 顾客信息录入
- 布料库管理
- 模板库管理
- 预览图生成
- 方案保存与历史查询
- 生成结果导出/微信发送准备

## 2. 里程碑

### M1：基础工程搭建

输出：

- 可运行的前后端项目
- 数据库和对象存储接通
- 登录和基础权限可用

### M2：素材与业务管理可用

输出：

- 布料库管理后台
- 模板库管理后台
- 顾客方案主流程打通

### M3：出图链路可用

输出：

- 模板映射
- AI 优化
- 生成任务状态追踪

### M4：门店试运行版本

输出：

- 生成结果页
- 历史方案查询
- 基础监控和错误处理

## 3. 技术分工建议

### 前端

- 门店操作端
- 后台素材管理端
- 结果展示与对比页

### 后端

- 业务 API
- 认证授权
- 数据库模型
- 文件上传与访问控制

### 图像/AI

- 布料预处理
- 模板贴图
- AI 优化编排
- 异步任务和失败重试

### 测试/运维

- 联调测试
- 性能测试
- 监控告警
- 试运行支持

## 4. 详细任务拆解

### 4.1 基础工程

#### 前端

- 初始化 `Next.js + TypeScript + Tailwind CSS`
- 建立门店操作端布局
- 建立后台管理端布局
- 接入基础 UI 组件库
- 配置环境变量和 API 请求封装

#### 后端

- 初始化 `NestJS` 或 `Next.js API` 工程
- 接入 `PostgreSQL + Prisma`
- 接入 `Redis`
- 接入对象存储 SDK
- 建立日志中间件和错误处理中间件

#### DevOps

- 配置开发环境
- 配置测试环境
- 配置 `.env` 模板
- 配置 CI 基础流程

### 4.2 认证与权限

#### 后端

- 用户登录接口
- JWT 或 Session 方案
- 角色权限中间件
- 登录态校验接口

#### 前端

- 登录页
- 登录态持久化
- 未登录拦截

一期只需：

- `admin`
- `manager`
- `clerk`

### 4.3 顾客管理模块

#### 后端

- 创建顾客接口
- 更新顾客接口
- 顾客详情接口
- 顾客搜索接口

#### 前端

- 顾客信息录入页
- 顾客搜索弹窗
- 顾客详情侧边栏

验收标准：

- 可按手机号快速定位顾客
- 可避免重复建档

### 4.4 布料库管理模块

#### 后端

- 创建布料接口
- 更新布料接口
- 布料列表接口
- 布料筛选接口
- 布料图片上传接口

#### 前端

- 布料列表页
- 布料新增/编辑弹窗
- 布料筛选区
- 布料详情预览

#### 数据准备

- 制定布料命名规则
- 制定布料图片采样规范
- 整理初始 50-100 款布料

验收标准：

- 能按颜色、材质、婚庆标签筛选
- 布料图片清晰可用于出图

### 4.5 模板库管理模块

#### 后端

- 创建模板接口
- 更新模板接口
- 模板列表接口
- 模板素材上传接口
- `mapping_config` 保存接口

#### 前端

- 模板列表页
- 模板编辑页
- 模板素材上传页
- 模板预览页

#### 图像工程

- 约定模板底图规范
- 约定遮罩图命名规范
- 设计 `mapping_config` JSON 格式

验收标准：

- 至少可管理 20-30 套模板
- 每套模板可关联底图和遮罩图

### 4.6 顾客方案模块

#### 后端

- 创建方案接口
- 更新方案接口
- 方案详情接口
- 历史方案列表接口
- 方案关联布料接口

#### 前端

- 新建方案页
- 选择布料页
- 选择模板页
- 方案概览页

验收标准：

- 店员能在 1 分钟内完成方案建档

### 4.7 图像预处理模块

#### 图像/AI

- 布料图片裁剪
- 布料图片缩略图生成
- 色彩归一化
- 简单纹理增强

#### 后端

- 提供预处理任务入口
- 保存预处理结果地址

验收标准：

- 布料图可稳定进入后续贴图流程

### 4.8 模板映射引擎

#### 图像/AI

- 读取模板底图和遮罩图
- 按区域执行布料平铺
- 执行透视变换
- 输出基础合成图

#### 后端

- 生成基础合成任务
- 保存基础图状态

验收标准：

- 能输出结构稳定的床品基础预览图
- 被套、床单、枕套区域不明显错位

### 4.9 AI 图像优化模块

#### 图像/AI

- 定义图像优化 prompt 模板
- 接入外部图像模型 API
- 生成正视图、侧视图、氛围图
- 支持“更喜庆/更高级/更简洁”等调优参数

#### 后端

- 统一 AI 调用接口层
- 保存 prompt 和返回结果
- 失败重试和超时控制

验收标准：

- 单次生成可返回 1-3 张可用于展示的图
- 失败时能回退到基础贴图图

### 4.10 任务队列与异步编排

#### 后端

- 建立 `BullMQ` 队列
- 任务入队接口
- 状态轮询接口
- 失败重试策略
- 超时和取消机制

#### 前端

- 生成中状态页
- 轮询或 WebSocket 更新
- 失败重试按钮

验收标准：

- 前端不会因出图阻塞
- 任务状态清晰可见

### 4.11 结果展示模块

#### 前端

- A/B/C 多方案展示
- 大图预览
- 收藏方案
- 顾客偏好标记

#### 后端

- 收藏接口
- 查询结果图接口

验收标准：

- 店员可快速给顾客做对比选择

### 4.12 历史方案与留存

#### 后端

- 历史方案分页接口
- 按时间/手机号/店员筛选
- 方案详情查询

#### 前端

- 历史方案列表页
- 筛选栏
- 详情回看页

验收标准：

- 能回看过去生成结果
- 能支撑后续顾客复购沟通

### 4.13 导出与微信发送准备

一期先做“导出可发送图片”，不要直接打通复杂微信接口。

#### 后端

- 分享图导出接口
- 带签名访问地址生成

#### 前端

- 下载结果图
- 复制图片链接
- 一键展示二维码或下载按钮

验收标准：

- 店员能把结果快速发给顾客

### 4.14 日志、监控与审计

#### 后端

- 操作日志记录
- 任务日志记录
- API 错误日志
- 慢任务告警

#### 运维

- 接入基础监控
- 接入错误告警
- 接入对象存储可用性检查

验收标准：

- 出问题时能定位是数据、模板还是 AI 链路问题

### 4.15 测试任务

#### 功能测试

- 顾客建档流程
- 布料筛选流程
- 模板选择流程
- 生成流程
- 历史方案回看

#### 异常测试

- 图片上传失败
- AI 接口超时
- 模板遮罩错误
- 对象存储不可用

#### 性能测试

- 单次生成耗时
- 并发 3-5 店员生成任务
- 热门模板缓存效果

## 6. 角色工作量建议

最小团队配置：

- 1 名前端工程师
- 1 名后端工程师
- 1 名图像/AI 工程师
- 1 名产品/项目协调

如果没有专职图像工程师，后端可以先兼做，但图像链路质量会直接决定成败。

## 7. 一期验收标准

必须满足：

- 顾客建档可用
- 布料库与模板库可管理
- 可稳定生成至少 1 张预览图
- 结果可保存、可回看
- 单次完整体验在门店可跑通

理想目标：

- 15-30 秒内生成 3 张图
- 可做 A/B/C 对比
- 失败任务可重试

## 8. 技术风险清单

- 模板映射不准，导致成品结构失真
- 布料采样不标准，导致颜色偏差
- AI 耗时过长，影响门店体验
- 前端交互太复杂，店员不愿用
- 任务链路缺监控，问题难定位

## 9. 开发顺序建议

不要先做最炫的 AI 优化，正确顺序是：

1. 先把布料、模板、方案三套数据结构做稳
2. 再把模板贴图跑通
3. 再接入 AI 优化
4. 最后做分享、监控和体验优化

因为如果基础数据层不稳，AI 只会放大问题。

## 10. 结论

这份任务拆解适合直接拿来做一期排期和分工。

如果要快速落地，优先级最高的模块只有 4 个：

- 布料库
- 模板库
- 方案流转
- 出图链路

这 4 个跑通，MVP 就成立了。
